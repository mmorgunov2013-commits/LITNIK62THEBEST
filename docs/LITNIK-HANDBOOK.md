# ЛИТНИК — справочник по проекту

Краткая памятка, если потерялся контекст диалога. Стек и правила из `AGENTS.md` / `CLAUDE.md` по-прежнему в силе.

## Что это за сайт

- **B2B-витрина** производителя гранул и сплавов из цветных металлов: каталог по категориям, карточки товаров, заявки, админка (CRUD товаров).
- **Стек:** Next.js (App Router), TypeScript, Tailwind, Prisma, PostgreSQL (часто Neon), Zod, `next/font` (Manrope), lucide-react.
- **Продакшен:** типично **GitHub → Render** (`next start`), база — отдельный PostgreSQL (в логах может быть `neon.tech`).

## Важные пути в коде

| Что | Где |
|-----|-----|
| Публичные страницы | `app/(site)/` |
| Конфиг сайта (тексты героя, плейсхолдер фото товара) | `lib/site.ts` |

## Категории и URL

- Slug категории **«Бронза»** в базе: **`bronza`** (латиница), не `bronze`.
- Редирект в `next.config.ts`: `/catalog/bronze` → `/catalog/bronza`.

## Карточка товара: откуда берутся данные

1. **Таблица `Product`** в PostgreSQL (Prisma: `prisma/schema.prisma`).
2. Поле **`image`** — одно фото (превью, совместимость).
3. Поле **`galleryImages`** — массив строк **URL** (порядок = порядок на сайте). Первая картинка в массиве = главное фото + превью в каталоге, если задана галерея.

Публичные файлы кладутся в **`public/`**, URL в браузере начинаются с **/**, например: `/images/products/имя-папки/01.jpg`.

## Как сделать порядок фото правильным

1. **Один товар — одна папка** под `public/images/products/<slug-папки>/` (например `brof-10-1`).
2. Файлы в картотеке: **`01.jpg`, `02.jpg`, … `06.jpg`** (две цифры, порядок слева направо, как в макете «Фото 1 … Фото 6»).
3. В **`galleryImages`** (админка или сид) перечисляй **те же пути в том же порядке**:
   ```text
   /images/products/brof-10-1/01.jpg
   /images/products/brof-10-1/02.jpg
   …
   ```
4. Если в исходниках файлы названы «Фото 1.jpg», «Фото 2.jpg» — при копировании в `public` **переименовывать в 01.jpg, 02.jpg**, чтобы порядок не путался с другими наборами («Броф N» и т.д.).
5. На странице товара: крупно первое фото, сетка миниатюр — **фото 2–6** (первое не дублируется в ряду).

## Как создать новую карточку товара

### Вариант А — админка (предпочтительно для продакшена)

1. Зайти в **`/admin`** (логин из `ADMIN_EMAIL` / `ADMIN_PASSWORD`, при первом сиде см. `prisma/seed.ts`).
2. Создать товар, выбрать категорию, **slug** латиницей через дефис.
3. Загрузить одно фото или указать **галерею**: в поле «Галерея» — **по одному URL с новой строки** (как выше).
4. Сохранить; при необходимости положить файлы в `public/...` заранее или через загрузку «Изображение» (один файл).

### Вариант Б — через JSON + полный seed

1. Добавить объект товара в **`data/litnik-products.source.json`** (поля как у соседних карточек).
2. Выполнить **`npx prisma db seed`**.  
   **Внимание:** текущий `seed.ts` **очищает** товары/категории/заявки и пересоздаёт — только для пустой/тестовой базы или осознанно.

### Вариант В — точечный скрипт (как у БроФ)

- Скопировать **`scripts/upsert-brof-product.ts`**, поменять slug, данные и массив `GALLERY`, добавить в `package.json` скрипт.
- Запуск локально: `npm run db:upsert-brof` (с тем же `DATABASE_URL`, что и прод).

## Как выкатить новые карточки и картинки на сервер (Render)

1. **На ПК:** положить новые файлы в **`public/...`**, закоммитить, **`git push`** в репозиторий, к которому привязан Render.
2. Render **пересоберёт** образ; статика из `public` попадёт в деплой.
3. **База:** если в коде только новые **картинки**, а **строки товара** уже созданы в админке — миграций может не быть.
4. Если добавляли **новую миграцию Prisma** — на сервере при билде обычно уже есть **`prisma migrate deploy`** (см. `package.json` build).
5. Если товар **только в базе** должен появиться без полного seed:
   - либо создать в **админке**,
   - либо запустить **одноразовый upsert-скрипт** в **Render Shell** (с тем же `DATABASE_URL`):
     ```bash
     npm run db:upsert-brof
     ```
     (или свой скрипт по аналогии).

**Полный `db seed` на проде** — только если осознанно готовы пересоздать каталог из JSON.

## Prisma и «Connection closed» (Neon)

- В логах может быть `neon.tech` — это **хост БД**, не отдельный продукт «вместо React».
- В `DATABASE_URL` для pooler у Neon часто добавляют параметры вроде **`sslmode=require`** и **`pgbouncer=true`** (см. доку Neon + Prisma).
- Краткий комментарий в **`lib/prisma.ts`**.

## Главная страница (герой)

- Фон задаётся **`siteConfig.heroBackgroundImage`** (файл в `public/images/`, обычно `hero-bg.jpg`).
- На главной блок **«Категории»** стоит в **той же секции**, что и заголовок/кнопки — **тот же фон гранул** (без отдельной чёрной полосы между ними). Отдельный `min-height` на весь экран не обязателен: высота от контента.
- Размер/позиция фона: **`.lit-hero-backdrop`** в `globals.css` (`cover`, на мобилке слегка смещение по вертикали).

## Чеклист: новый товар с 6 фото

1. Подготовить **`01.jpg`…`06.jpg`** в `public/images/products/<папка>/`.
2. Админка → галерея: шесть строк URL по порядку **или** правка JSON + seed/скрипт.
3. `git add`, `commit`, `push`.
4. Убедиться, что на Render прошёл деплой; при необходимости один раз запустить upsert/создать товар в админке.
